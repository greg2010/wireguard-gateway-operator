# Operator pivot — wireguard-gateway-operator

Plan of record for converting this repo from a Helm-templated Crossplane tool into a
controller-runtime **operator**. Temp doc; delete after the pivot lands.

## Goal

The end user creates a `Gateway` CR. The operator reconciles it into:
a Crossplane `XGateway` composite (provisions the GCP gateway VM), an in-cluster
WireGuard **link** Deployment that dials that gateway, the WG key Secrets, and an
optional DNSEndpoint — and watches for changes.

```
[user] kubectl apply Gateway
   -> [gateway-operator reconcile] ensures, per Gateway CR (owner-ref'd children):
        1. WG key Secrets        (generate once, never rotate; beacon + link keypairs -> bundle Secret + link Secret)
        2. XGateway composite    (built via the regenerated typed `XGatewaySpec` — operator is its first Go consumer; provisions GCP; carries wgKeySecretRef -> bundle Secret + derived serviceAccountId/secretId. `serviceAccountEmail` is NOT set here — it is GCP-observed, written by the composition from the ServiceAccount MR's status.atProvider.email and surfaced via the composite status)
        3. link Deployment+CM+RBAC (gateway-link; reads XGateway.status.address, self-heals)
        4. DNSEndpoint           (externaldns.k8s.io) once the IP is known, if dnsHostnames set
   -> watches Gateway + Owns(XGateway [IP-change requeue], Deployment, CM, Secret, SA, Role, RB)
   -> status: address + Ready condition
```

Layering: `Gateway` (user API) -> operator -> `XGateway` (composite) -> Crossplane -> GCP.
The XGateway composite, the link binary, and xrdgen are **kept**; the regenerated typed `XGatewaySpec` gains its first Go consumer (the operator's XGateway builder — the link + harness read the composite via dynamic/unstructured paths, not the typed pkg).

## Key design resolutions

- **WG params are operator-global** (subnet/addresses/listenPort/keepalive/mtu come from chart values -> one static Flatcar Ignition `userData`). "Static `userData`" means the operator-global boot fetch script bakes **no** per-Gateway value: the per-Gateway `secretId` reaches the VM via **GCP instance metadata** (`XGateway.spec.secretId` -> Instance MR `metadata` -> `keyfetch.sh` reads it from the metadata server at boot), not via `userData`. The WG params `keyfetch.sh` still references (WG listen port, link address) are operator-global, so the rendered `userData` is genuinely identical across every Gateway. The per-Gateway WG keypair is **not** in `userData` either; it reaches the VM via `XGateway.spec.wgKeySecretRef` -> GCP Secret Manager (at the metadata-supplied `secretId`) -> the boot fetch script. The user's `Gateway` spec carries only gcp/forwards/dnsHostnames; the operator sets `wgKeySecretRef` + the derived GCP IDs on the `XGateway` composite, not on the user CR. (Per-Gateway WG params would require per-Gateway Ignition rendering — deferred.)
- **Unique GCP IDs per Gateway**: the operator derives `serviceAccountId`/`secretId` as `gw-<lowercase-base32(sha256(namespace + "/" + name))>`, truncated to GCP's 30-char service-account-ID limit (GCP SA/Secret IDs are project-global; the shared default collides across tenants). The `gw-` prefix guarantees a leading letter, base32's `[a-z2-7]` charset is GCP-valid, and the namespace-qualified input is collision-resistant across namespaces. The XGateway name drives the network/subnet/firewall/instance GCP names (composition derives them from the MR name). The operator derives only the SA/Secret **IDs**; `serviceAccountEmail` is **GCP-observed** — the composition writes it from the ServiceAccount MR's `status.atProvider.email` and it flows up via the composite status, so the operator never computes the email.
- **`Owns(XGateway)`** drives the address-change requeue; **do not** `Owns(DNSEndpoint)` (would make the manager start depend on the external-dns CRD being installed).
- **Teardown is Gateway-delete-driven**: a finalizer on `Gateway` drives the ordered GCP drain (see below), so `kubectl delete gateway` blocks until the cloud is drained; owner-ref GC then cascades link + Secrets + DNS. The namespace must outlive the drain (ProviderConfigUsage is per-namespace); the Gateway CRD is removed last.
- **Finalizer on Gateway drives ordered teardown**: on delete the reconciler deletes the `XGateway` and requeues until it is gone (Crossplane drains GCP), then releases the finalizer. Owner-ref GC still cascades the link/Secrets/DNS. The `XGateway` is the only child **not** owner-ref-reaped — it is finalizer-deleted explicitly — so `Owns(XGateway)` drives requeue, not GC removal.

Tests** — e2e: rework `test/harness/e2e/{stack.go,values.go}` to apply a `Gateway` CR and poll **its** `status.address` (the current harness does a single `helm install --wait` then one `GetXBeaconStatus` read — it does not poll a provision Job; the rework replaces that single post-install status read with applying a `Gateway` CR and polling `status.address`), and drop the dead overlay fields: the `beacon.create` SetValue (absent from `values.yaml`) and the `SecretID` field in `values.go` (set to `namePrefix+"-wg"`) — `gcp.secretId` is operator-derived now and gone from the chart, so the overlay setting it is dead config. Install the operator chart, create the `Gateway` CR, assert `status.address` + positive tcp/udp data-path probes **+ negative probes** (a non-forwarded public port must NOT reach the pod, tcp and udp). A non-forwarded port is dropped at the GCP firewall (the composition opens only `allowedPorts` + the WG listen port), so the negative probe validates firewall + DNAT closure jointly, not nftables alone. The UDP negative must be timeout-based — agnhost's UDP server replies only to an `echo` command, so "no reply within a timeout" is the pass signal (no ICMP "connection refused" to rely on). The negative probe uses a SHORT fixed timeout asserting non-delivery — distinct from the positive `dataPathDeadline` (3m retry-until-success); do **not** reuse `retryUntil` for the negative case (it would wait the full deadline). Pin a concrete non-forwarded probe port (e.g. `9999`); the harness must **FAIL setup** if that port appears in `forwards[]` or equals the WG listen port — a Done-criterion, not just prose, so a future `forwards[]` change can't silently turn the negative probe into a false pass. All three are deleted because DNSEndpoint ownership moves to the operator. DNSEndpoint creation is **unimplemented on disk**: the provision-Job's `provision` mode is a chart template whose env (`CYNO_BOOTSTRAP_MODE=provision`, `CYNO_DNS_ENDPOINT_NAME`, …) `cyno-bootstrap` ignores (`cyno-bootstrap` only does keygen). So Phase D's DNS builder is **new** work, not a port. Deleting the present-asserting case is a correctness fix; Phase H's `make test-integration` is not green until it is removed. Unit: controller envtest + builder table tests. **Done** = e2e applies a `Gateway` CR, polls `Gateway.status.address`, asserts positive tcp/udp + timeout-based negative tcp/udp probes; the harness fails setup if the negative-probe port (`9999`) collides with `forwards[]` or the WG listen port; `composition_test.go` DNS cases removed; controller envtest + builder table tests green.

## Risks / watch-items

- controller-runtime/controller-tools vs k8s 0.35.4 (resolved: v0.23.3 / v0.20.1; pins unchanged).
- GCP SA/Secret ID global uniqueness -> derive per-Gateway via the `gw-<base32(sha256(ns/name))>` rule (resolution above).
- Crossplane v2 must preserve the operator's owner-ref on the namespaced XGateway (verify).
- `Owns(unstructured XGateway)` needs the xgateways CRD RESTMappable at manager start -> Crossplane is a hard install prereq; external-dns provides the DNSEndpoint CRD (prereq for the DNS feature).
- envtest assets out-of-module (`setup-envtest@v0.24.1`, k8s 1.35.x); `KUBEBUILDER_ASSETS` for `make test-unit`.
- **Teardown drain ordering (HARD constraint)**: delete all `Gateway`s and wait for the GCP drain BEFORE `helm uninstall` / namespace deletion. Force-deleting the namespace bypasses the finalizer and orphans GCP resources. Order: Gateway delete -> wait GCP drain (namespace alive) -> uninstall operator -> delete namespace; CRD removed last.
- Operator ClusterRole is controller-gen-generated from the full `//+kubebuilder:rbac` marker set in `internal/controller` (`secrets`, `xgateways`, `gateways`/`status`/`finalizers`, `dnsendpoints`, plus leader-election `leases` + `events`) — not just the leader-election verbs; `make manifests` must emit it complete.
- Static `userData` requires WG params to be operator-global (design resolution above).
